    **Part A: Branch coverage report (AUDT-03)**

    Run pytest with branch coverage on the cleaned test suite (milestone-markers already deleted by Plan 01):
    ```bash
    poetry run pytest \
      --cov=bitbat \
      --cov-branch \
      --cov-report=term-missing \
      -q \
      2>&1 | tee .planning/phases/24-audit-baseline/evidence/coverage.txt
    ```

    Do NOT generate HTML coverage report (unnecessary for audit evidence). The `term-missing` output provides module-by-module percentages with missing line numbers.

    After the run, extract from coverage.txt a ranked list of the lowest-coverage modules (bottom 10). Per research, expected low modules include:
    - `api/routes/system.py` (~46%)
    - `autonomous/continuous_trainer.py` (~47%)
    - `analytics/feature_analysis.py` (~65%)
    - `gui/widgets.py` (~69%)

    Note: Coverage numbers may shift slightly from the research pre-run because milestone-marker tests have been deleted. This is expected and desirable — it shows true behavioral coverage.

    **Part B: E2E pipeline smoke test (AUDT-05)**

    Execute the pipeline stages sequentially per CONTEXT.md decisions:
    1. Try real data first (yfinance small date range), fall back to synthetic if network fails
    2. Skip-and-continue on failures: if a stage fails, try next stage with pre-existing data
    3. Save raw console log and structured summary

    **Create a smoke test config file first:**
    ```bash
    mkdir -p /tmp/smoke_test/data/raw/prices
    cat > /tmp/smoke_config.yaml << 'EOF'
    data_dir: "/tmp/smoke_test/data"
    freq: "1h"
    horizon: "4h"
    tau: 0.005
    enable_sentiment: false
    enable_garch: false
    enable_macro: false
    enable_onchain: false
    seed: 42
    EOF
    ```

    Use `--config /tmp/smoke_config.yaml` for all stages. The config disables auxiliary features (sentiment, GARCH, macro, on-chain) to reduce external dependencies during the smoke test. Use `1h`/`4h` (the project default pair) rather than `5m`/`30m` from research, since the default config is what operators actually run.

    **Execute each stage, capturing output:**

    Stage 1 - Ingest prices:
    ```bash
    poetry run bitbat --config /tmp/smoke_config.yaml prices pull --symbol BTC-USD --start 2025-01-01 --interval 1h --output /tmp/smoke_test/data/raw/prices 2>&1 | tee /tmp/smoke_test/stage1.log
    ```
    If this fails (network, API), note the failure and check if `data/raw/prices/` in the project root has existing data to copy.

    Stage 2 - Build features:
    ```bash
    poetry run bitbat --config /tmp/smoke_config.yaml features build 2>&1 | tee /tmp/smoke_test/stage2.log
    ```

    Stage 3 - Train model:
    ```bash
    poetry run bitbat --config /tmp/smoke_config.yaml model train 2>&1 | tee /tmp/smoke_test/stage3.log
    ```

    Stage 4 - Batch prediction:
    ```bash
    poetry run bitbat --config /tmp/smoke_config.yaml batch run 2>&1 | tee /tmp/smoke_test/stage4.log
    ```

    Stage 5 - Monitor run-once:
    ```bash
    poetry run bitbat --config /tmp/smoke_config.yaml monitor run-once 2>&1 | tee /tmp/smoke_test/stage5.log
    ```

    **Concatenate all stage logs:**
    ```bash
    cat /tmp/smoke_test/stage*.log > .planning/phases/24-audit-baseline/evidence/smoke-test.log
    ```

    **Produce smoke-summary.md:**
    Write `.planning/phases/24-audit-baseline/evidence/smoke-summary.md` with:
    - Table: `| Stage | Command | Result | Error (if any) |`
    - For each failed stage: exact error message, traceback excerpt, and which requirement it maps to
    - For each passed stage: confirmation it produced expected output files
    - Note any stages where fallback to existing data was used
    - Note the CORR-01 bug context: `model train` will succeed when called directly, but fails when called by the autonomous retrainer with `--tau`. This is NOT testable in the smoke test — document it as a known gap.
