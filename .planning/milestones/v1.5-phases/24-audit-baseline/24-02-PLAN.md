<task type="auto">
  <name>Task 1: Vulture dead code scan with whitelist triage</name>
  <files>
    .planning/phases/24-audit-baseline/evidence/vulture.txt
    .planning/phases/24-audit-baseline/evidence/vulture-whitelist.py
    .planning/phases/24-audit-baseline/evidence/vulture-triaged.txt
  </files>
  <action>
    **Step 1: Run raw vulture scan**

    Run vulture at 80% confidence, scanning `src/bitbat` as the target and `tests/` as usage context (so test imports count as "used"):
    ```bash
    poetry run vulture src/bitbat tests/ --min-confidence 80 > .planning/phases/24-audit-baseline/evidence/vulture.txt 2>&1
    ```
    Save the full raw output.

    **Step 2: Generate and curate whitelist**

    Run `--make-whitelist` on src/bitbat only:
    ```bash
    poetry run vulture src/bitbat --min-confidence 80 --make-whitelist > .planning/phases/24-audit-baseline/evidence/vulture-whitelist.py 2>&1
    ```

    Then manually curate the whitelist. Review each item and KEEP in the whitelist only confirmed false positives:
    - Pytest fixture parameters (used via dependency injection): `db_with_data`, `db_with_predictions`, `model_on_disk`, etc.
    - Click command/option decorators (vulture can't trace decorator registration)
    - `__all__` exports in `__init__.py` files
    - FastAPI route functions registered via decorators
    - Pydantic model fields used by serialization

    REMOVE from the whitelist any items that appear to be genuinely dead code (unused functions, unused imports, unused variables that are not fixtures).

    **Step 3: Re-run with whitelist applied**

    ```bash
    poetry run vulture src/bitbat tests/ .planning/phases/24-audit-baseline/evidence/vulture-whitelist.py --min-confidence 80 > .planning/phases/24-audit-baseline/evidence/vulture-triaged.txt 2>&1
    ```

    This produces the triaged output showing only genuine dead code candidates.

    **Step 4: Annotate triaged findings**

    Add a comment block at the top of `vulture-triaged.txt` with:
    - Total raw findings count vs triaged count
    - For each remaining finding, note which v1.5 requirement it maps to (if any), or 'DEFER' if it is genuine dead code but out of scope for v1.5
    - Flag any finding that maps to CORR-01 through ARCH-06
  </action>
