<objective>
Wire the API settings endpoint to default.yaml as its fallback source and validate freq/horizon values against bucket.py's canonical frequency set, so operators get correct defaults and can persist sub-hourly configurations.

Purpose: The current settings endpoint falls back to the "balanced" preset (1h/4h) instead of default.yaml (5m/30m). This mismatch means operators see wrong defaults and sub-hourly values are neither validated nor reliably persisted. Phase 21 needs valid option lists from this endpoint to build dynamic dropdowns.

Output: Settings GET/PUT handlers backed by default.yaml with frequency validation, schema with valid option lists, and regression tests.
</objective>

Rewrite the GET and PUT `/system/settings` handlers in `src/bitbat/api/routes/system.py`:

**GET handler changes:**
1. Import `load_config` from `bitbat.config.loader` and `_SUPPORTED_FREQUENCIES` from `bitbat.timealign.bucket`.
2. Remove the `from bitbat.gui.presets import get_preset` fallback. The fallback source is now `load_config()` (which reads default.yaml).
3. When no user config exists (`_USER_CONFIG_PATH` doesn't exist): load defaults via `load_config()` and return `SettingsResponse(preset="custom", freq=defaults["freq"], horizon=defaults["horizon"], tau=defaults.get("tau", 0.01), enter_threshold=defaults.get("enter_threshold", 0.6), valid_freqs=sorted_freqs, valid_horizons=sorted_horizons)`.
4. When user config exists: read it, merge missing fields with `load_config()` defaults (per user decision: "Missing user-config fields fall back to default.yaml values").
5. Compute `valid_freqs` and `valid_horizons` from `_SUPPORTED_FREQUENCIES`. For valid_freqs, use the full set sorted by duration: `["1m", "5m", "15m", "30m", "1h", "4h", "24h"]`. For valid_horizons, use the same set minus "1m" (a 1-minute horizon is impractical): `["5m", "15m", "30m", "1h", "4h", "24h"]`. Sort by converting to pandas Timedelta for correct ordering. Add a helper `_sorted_by_duration(freqs: set[str]) -> list[str]` that sorts frequency strings by their actual time duration.
6. Always include `valid_freqs` and `valid_horizons` in every response.

**PUT handler changes:**
1. Validate `request.freq` (if provided) against `_SUPPORTED_FREQUENCIES`. If not in the set, raise `HTTPException(status_code=422, detail=f"Unsupported frequency '{request.freq}'. Valid: {sorted_freqs}")`.
2. Validate `request.horizon` (if provided) against `_SUPPORTED_FREQUENCIES` minus "1m". Same error pattern.
3. Keep partial merge behavior: start from current GET result as base, overlay only provided fields.
4. When `request.preset` is provided, still resolve it via `get_preset` for backward compatibility — but if `request.freq` or `request.horizon` is also provided, those override the preset values.
5. Write the merged dict to `_USER_CONFIG_PATH` (YAML, same as current behavior).
6. Return merged result with `valid_freqs` and `valid_horizons`.

**Config loader change (minimal):**
In `src/bitbat/config/loader.py`, no functional changes needed — `load_config()` already reads default.yaml when no path is given. The route handler will call `load_config()` directly.
