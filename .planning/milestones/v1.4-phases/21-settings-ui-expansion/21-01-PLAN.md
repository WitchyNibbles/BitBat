**Changes required:**

1. **Remove hardcoded freq/horizon from DEFAULTS.** Change the DEFAULTS constant to only contain values that the API does NOT provide (tau and confidence have UI-specific transformations, so keep fallbacks for those). Remove `freq` and `horizon` from DEFAULTS since the API provides both the current values AND the valid options:

```typescript
const DEFAULTS = {
  preset: 'balanced',
  tau: 0.005,
  confidence: 70,
};
```

2. **Initialize freq and horizon state as empty strings** (they will be populated from the API response):

```typescript
const [freq, setFreq] = useState('');
const [horizon, setHorizon] = useState('');
```

3. **Update the useEffect that populates from loaded settings** to always use API values for freq and horizon (no fallback to hardcoded defaults):

```typescript
useEffect(() => {
  if (settings.data) {
    setPreset(settings.data.preset || DEFAULTS.preset);
    setFreq(settings.data.freq);
    setHorizon(settings.data.horizon);
    setTau(settings.data.tau ?? DEFAULTS.tau);
    setConfidence(
      settings.data.enter_threshold != null
        ? Math.round(settings.data.enter_threshold * 100)
        : DEFAULTS.confidence,
    );
  }
}, [settings.data]);
```

4. **Replace hardcoded `<option>` elements with dynamic rendering** from `settings.data.valid_freqs` and `settings.data.valid_horizons`. The frequency dropdown:

```tsx
<select
  className={styles.select}
  value={freq}
  onChange={(e) => setFreq(e.target.value)}
>
  {(settings.data?.valid_freqs ?? []).map((f) => (
    <option key={f} value={f}>{f}</option>
  ))}
</select>
```

The horizon dropdown:

```tsx
<select
  className={styles.select}
  value={horizon}
  onChange={(e) => setHorizon(e.target.value)}
>
  {(settings.data?.valid_horizons ?? []).map((h) => (
    <option key={h} value={h}>{h}</option>
  ))}
</select>
```

5. **Update handleSave** — freq and horizon should always be sent (remove the `showAdvanced` conditional for freq/horizon since the dropdowns always reflect the current API-persisted values). The save handler becomes:

```typescript
const handleSave = useCallback(async () => {
  setSaving(true);
  setStatusMsg('');
  setErrorMsg('');

  try {
    await api.updateSettings({
      preset,
      freq: showAdvanced ? freq : null,
      horizon: showAdvanced ? horizon : null,
      tau: showAdvanced ? tau : null,
      enter_threshold: showAdvanced ? confidence / 100 : null,
    });
    setStatusMsg('Settings saved successfully.');
  } catch (err) {
    setErrorMsg(err instanceof Error ? err.message : String(err));
  } finally {
    setSaving(false);
  }
}, [preset, freq, horizon, tau, confidence, showAdvanced]);
```

Note: Keep the `showAdvanced` conditional — freq/horizon only save when advanced settings are visible. This matches the existing dashboard save pattern (per Claude's discretion in CONTEXT.md: "Auto-save on dropdown change vs explicit Save button — follow existing dashboard save patterns").

6. **Update handleReset** to reload from the API instead of hardcoded defaults. Replace the direct state reset with a refetch:

```typescript
const handleReset = useCallback(() => {
  settings.refetch();
  setStatusMsg('');
  setErrorMsg('');
}, [settings]);
```

This fetches fresh defaults from the API (which returns default.yaml values when no user config exists). The useEffect will repopulate state from the response. This is cleaner than hardcoding reset values that may not match the actual defaults.
